# Release & Changelog

## How releases work

1. **tagpr** creates a release PR automatically when changes land on `main`
2. Version bump is determined by PR labels:
   - `release/major` or `kind/breaking-change` → major
   - `release/minor`, `kind/feature`, or `kind/deprecation` → minor
   - Everything else → patch
3. tagpr updates `CHANGELOG.md` and `package.json` version, then creates a git tag on merge
4. The publish workflow (`.github/workflows/publish.yaml`) builds and pushes the Docker image on tag push

## Changelog sources

- **`CHANGELOG.md`** — Generated by tagpr. Lists merged PR titles grouped by version. PRs with `changelog/skip` label are excluded.
- **GitHub Release notes** — Generated by GitHub using `.github/release.yml`. Categorizes PRs by `kind/*` labels (Breaking Changes, New Features, Bug fixes, Improvements, etc.). PRs without a `kind/*` label fall into "Others".

## Label flow

```
PR merged → tagpr reads labels → determines version bump → updates CHANGELOG.md
         → GitHub Release created → .github/release.yml categorizes by kind/* labels
```

## Key files

| File | Purpose |
|------|---------|
| `.tagpr` | tagpr config: release branch, version file, label-to-version mapping |
| `.github/release.yml` | GitHub Release notes: label-to-category mapping |
| `.github/label-definitions.yaml` | Source of truth for all repository labels (synced by label-sync workflow) |
| `.github/workflows/tagpr.yaml` | tagpr workflow |
| `.github/workflows/publish.yaml` | Docker image publish on tag |
